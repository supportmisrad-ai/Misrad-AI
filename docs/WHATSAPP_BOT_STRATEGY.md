## 1. מחקר שוק

### 1.1 הנתונים המספריים (2025-2026)

| מדד | ערך | מקור |
|------|------|------|
| שיעור פתיחת הודעות וואטסאפ | **98%** | Flowcart, LeadGen Economy |
| שיעור פתיחת מיילים (לשם השוואה) | 20-25% | ממוצע ענפי |
| שיעור תגובה בוואטסאפ | **45-60%** | Flowcart |
| שיעור המרת לידים עם צ'אטבוט | **28%** | LeadGen Economy |
| עלייה ביצירת לידים עם בוט | **+500%** | LeadGen Economy |
| ירידה בזמן טיפול בליד | **-61%** | LeadGen Economy |

### 1.2 אסטרטגיות מוכחות שחברות משתמשות בהן

**א. Click-to-WhatsApp Ads (CTWA)**
- המודל שמנצח: במקום טופס באתר → מודעה בפייסבוק/אינסטגרם שפותחת ישירות שיחה בוואטסאפ
- מבטל חיכוך: אין landing page, אין טופס, אין זמני טעינה
- תוצאה: קפיצה של 3-5x בהמרות לעומת טפסים רגילים
- **רלוונטי ל-MISRAD AI**: מודעות שמקדמות התנסות חינם / הדגמה → נפתח צ'אט עם הבוט

**ב. רצפי שיחה אוטומטיים (Conversational Sequences)**
- ברגע שליד נכנס — ברכה מיידית + 1-3 שאלות סיווג
- תיוג אוטומטי: `[גודל-ארגון]`, `[תחום-עניין]`, `[רמת-דחיפות]`
- העברה לנציג אנושי רק כשהליד "חם" — חוסך 80% מהזמן של צוות המכירות
- **חסר אצלנו**: הסיווג שלנו שטחי — שואלים גודל ארגון אבל לא תעשייה, לא pain point

**ג. רצפי nurturing (טיפוח)**
- **יום 1**: תודה + סיכום ערך
- **יום 3**: תוכן רלוונטי (מאמר/סרטון)
- **יום 5**: case study / testimonial
- **יום 7**: הצעה ישירה / הנחה
- **חסר אצלנו לחלוטין**: אין רצף nurturing, אין follow-up אוטומטי

**ד. קמפייני re-engagement**
- לידים שנשרו → הודעה אחרי 7/14/30 יום
- "היי {שם}, מאז שדיברנו יש לנו {פיצ'ר חדש}. רוצה לשמוע?"
- **חסר אצלנו**: אין שום מנגנון re-engagement

**ה. אינטגרציית CRM**
- כל פיסת מידע מהבוט → ישירות ל-CRM
- attribution מלא: מאיזו מודעה הגיע הליד, מה אמר, מה בחר
- **חסר אצלנו**: Webhooks מצביעים ל-Make.com placeholders — לא לבקנד שלנו

### 1.3 טעויות נפוצות שצריך להימנע מהן

1. **שליחת הודעות ללא הסכמה** — בישראל חוק הספאם אוסר
2. **אותה הודעה לכולם** — חייבים פרסונליזציה
3. **הצפה בהודעות** — 1-2 הודעות follow-up מקסימום
4. **תגובה איטית** — מעל 5 דקות = ירידה של 80% בהמרה
5. **התעלמות מאנליטיקס** — חייבים לעקוב מה עובד
6. **חשבון אישי במקום עסקי** — חובה WhatsApp Business

## 2. ניתוח הבוט הנוכחי

### 2.1 מה בניתי (ובעיות בו)

הבוט הנוכחי (`scripts/whatsapp-bot-rules.cjs`) מכיל 65 כללים ב-10 סקשנים. **הבעיות:**

| בעיה | חומרה | פירוט |
|-------|--------|--------|
| **פורמט .rules לא נטען** | קריטי | ⚠️ הקובץ עדיין לא עולה ב-Blaster — ייתכן שצריך להיבנות ישירות בתוכנה |
| **טון גנרי מדי** | גבוה | נשמע כמו תרגום מאנגלית, לא ישראלי |
| **אין הומור** | בינוני | בוט ישראלי חייב קצת הומור, במיוחד בפתיחה |
| **אין segmentation לפי תעשייה** | גבוה | לא שואלים מה התעשייה — לא יכולים להתאים |
| **אין social proof** | גבוה | אין testimonials, אין מספרים, אין case studies |
| **אין nurturing sequence** | גבוה | ליד שנרשם ולא סיים → נעלם |
| **אין re-engagement** | בינוני | לידים ישנים = כסף על הרצפה |
| **Webhooks מצביעים ל-Make** | בינוני | תלות בשירות צד שלישי + עלות |
| **אין מנגנון opt-out** | קריטי | חובה חוקית בישראל |
| **אין שעות פעילות/שבת** | גבוה | בוט שעונה בשבת = בעיה לקהל מסוים |

### 2.2 מה עובד

- מבנה הכללים (סקשנים, anchors, variables) — תקין טכנית
- זרימת ההרשמה (4 שלבים) — סבירה
- שאלות נפוצות — טוב אבל צריך יותר
- מודולים — המידע נכון ומעודכן לפי `pricing.ts`

## 3. לקחים מהפרויקט הישן (מעשיך יקרבוך)

מהפרויקט הישן (93 כללים, 22 מסמכי תיעוד) למדתי:

### 3.1 מה עבד שם

- **זרימת משתנים מורכבת**: איסוף העדפות לפי קטגוריות → התאמה אישית
- **מבנה "עצים"**: תת-כללים עם `follow_index` יוצרים עצי שיחה עמוקים
- **אינטגרציית Make + Google Sheets**: נתוני משתמשים זורמים לגיליון → מעובדים
- **פרופילים מרובים**: כל פרופיל = בוט שלם עם כללים משלו

### 3.2 מה נכשל שם (מהמסמך "מסקנות ותובנות")

- **סיבוכיות יתר**: 93 כללים עם תלויות מורכבות → קשה לתחזק
- **תלות במייק**: כל webhook = תרחיש במייק = עלות + נקודת כשל
- **חוסר גמישות**: שינוי בזרימה = שינוי ב-10 כללים
- **אין ניטור**: אין דרך לדעת איפה לידים נתקעים

### 3.3 מסקנות ליישום

> **כלל ברזל**: פשוט > מורכב. עדיף 40 כללים חכמים מ-100 כללים מסובכים.
> **כלל שני**: כל webhook צריך לזרום לבקנד שלנו, לא דרך Make.
> **כלל שלישי**: כל שינוי בבוט צריך להיות ניתן לעשות מהאדמין — לא מהקובץ.

## 4. אסטרטגיה חדשה ל-MISRAD AI

### 4.1 מטרות הבוט

| מטרה | KPI | יעד |
|-------|-----|-----|
| **לידים חדשים** | הרשמות לנסיון חינם | 50/חודש |
| **הדגמות** | הדגמות שנקבעו | 20/חודש |
| **סיווג לידים** | לידים מסווגים (תעשייה + גודל + צורך) | 80% |
| **תמיכה** | פניות שנפתרו אוטומטית | 60% |
| **שימור** | לקוחות קיימים שחזרו לבוט | 30%/חודש |

### 4.2 קהלי יעד

| סגמנט | מאפיינים | צורך עיקרי | חבילה מתאימה |
|--------|-----------|-------------|--------------|
| **עצמאי / פרילנסר** | עובד לבד, 0-1 עובדים | סדר בלקוחות או מכירות | Solo (149₪) |
| **עסק קטן** | 2-10 עובדים, צוות אחד | ניהול צוות + מכירות OR שיווק | Closer / Authority (249-349₪) |
| **חברה בינונית** | 11-50 עובדים, כמה צוותים | ניהול מלא, דשבורד | Empire (499₪) |
| **ארגון גדול** | 51-200 עובדים | כל המודולים + התאמה | Empire + מושבים (499₪+) |
| **לקוח קיים** | כבר משתמש | תמיכה, שדרוג, חידוש | — |

### 4.3 עקרונות ליבה

1. **שיחה, לא טופס** — כל אינטראקציה צריכה להרגיש כמו שיחה עם חבר שמבין בתוכנה
2. **ישראלי עד הסוף** — עברית טבעית, ביטויים מהרחוב, קצת הומור
3. **מהיר ל-value** — תוך 3 הודעות הליד צריך להבין למה MISRAD AI רלוונטי *לו*
4. **AI = יד בלתי נראית** — לא מדברים על "AI מדהים", אלא על "המערכת אומרת לך מה לעשות"
5. **נציג אנושי = רשת ביטחון** — תמיד אפשר לדבר עם בן אדם, והעברה חלקה
6. **Opt-out ברור** — "הקלד סטופ בכל שלב" — חובה חוקית ומוסרית

## 6. טון דיבור וכללי תקשורת

### 6.1 עקרונות

- **ישראלי אותנטי**: "מה קורה?" לא "שלום וברכה"
- **ישיר**: תוך 2 שורות להגיד את הנקודה
- **אנושי**: לפעמים אימוג'י, לפעמים בדיחה קלה
- **מקצועי**: כשמדובר בכסף/חיוב — רציני ומדויק
- **לא מתנשא**: "בוא נבדוק ביחד" לא "אתה צריך את זה"

### 6.2 דוגמאות טון

**פתיחה — לפני:**
> היי! אני הבוט של MISRAD AI. מערכת ה-AI שמנהלת את הארגון שלך...

**פתיחה — אחרי:**
> מה קורה! 👋 אני כאן מ-MISRAD AI.
> בקצרה? מערכת שאומרת לך מה לעשות במקום שתחפש לבד.
> מה מעניין אותך?

**חבילה — לפני:**
> חבילת The Closer — מכירות 💼. כל מה שצריך לסגור יותר עסקאות.

**חבילה — אחרי:**
> *The Closer* — לאנשי מכירות שעייפו לרדוף אחרי לידים 💼
> המערכת תגיד לך: "הליד הזה? 75% סיכוי. תתקשר עכשיו."
> 249₪/חודש, 5 אנשי צוות. 7 ימי ניסיון חינם.

**FAQ — לפני:**
> ההבדל? AI שעובד בשבילך. CRM רגיל = כלי שאתה מזין אליו נתונים.

**FAQ — אחרי:**
> CRM רגיל = אתה מקליד, המערכת שומרת. 🥱
> MISRAD AI = המערכת מקלידה לך ואומרת "תתקשר לדני, הוא עומד לברוח."
> זה ההבדל. בדוק בעצמך — 7 ימים חינם.

### 6.3 כללי "עשה" ו"אל תעשה"

## 8. אינטגרציית API — MISRAD AI ↔ Blaster

### 8.1 מצב נוכחי: Make.com (בעייתי)

```
[Blaster] → Webhook → [Make.com] → [Google Sheets / Email / CRM]
                           ↑
                     עלות, latency, נקודת כשל
```

### 8.2 הצעה: Webhook ישיר ל-MISRAD AI

```
[Blaster] → Webhook → [MISRAD AI Backend] → [Prisma DB]
                              ↓
                     [Admin Panel — ניהול מלא]
```

**מה זה נותן:**
- ✅ **אפס עלות** — לא צריך Make.com
- ✅ **אפס latency** — ישירות לשרת שלנו
- ✅ **מידע ב-DB שלנו** — Prisma, לא Google Sheets
- ✅ **שליטה מלאה** — Admin panel לצפייה בלידים, שיחות, משתנים
- ✅ **אנליטיקס** — כמה לידים, מאיפה, מה בחרו, איפה נשרו

### 8.3 ארכיטקטורה טכנית

#### Webhook Receiver (API Route)

```
POST /api/webhooks/blaster
```

**מה Blaster שולח ב-Webhook:**
- מספר טלפון של הפונה
- שם (אם נשמר במשתנה)
- תוכן ההודעה
- משתנים שנאספו (USER_NAME, BUSINESS_NAME, USER_EMAIL, וכו')
- Rule ID שהופעל

**מה נעשה עם המידע:**
1. **שמירה בטבלת `bot_leads`** — ליד חדש או עדכון קיים
2. **שמירה בטבלת `bot_conversations`** — היסטוריית שיחה
3. **יצירת BusinessClient** — אם הליד נרשם לנסיון
4. **שליחת notification** — לאדמין/נציג מכירות
5. **Analytics** — עדכון counters

#### מודל DB (Prisma)

```prisma
model BotLead {
  id              String   @id @default(cuid())
  phone           String   @unique
  name            String?
  businessName    String?
  email           String?
  industry        String?  // תעשייה
  orgSize         String?  // גודל ארגון
  painPoint       String?  // pain point
  selectedPlan    String?  // חבילה שנבחרה
  source          String?  // CTWA, QR, direct
  status          String   @default("new") // new, qualified, demo_booked, trial, customer, churned
  score           Int      @default(0)
  lastInteraction DateTime @default(now())
  createdAt       DateTime @default(now())
  updatedAt       DateTime @updatedAt
  conversations   BotConversation[]
  organizationId  String?
  organization    Organization? @relation(fields: [organizationId], references: [id])
}

model BotConversation {
  id        String   @id @default(cuid())
  leadId    String
  lead      BotLead  @relation(fields: [leadId], references: [id])
  direction String   // "in" | "out"
  message   String
  ruleId    String?
  variables Json?
  createdAt DateTime @default(now())
}
```

#### Admin Panel

| עמוד | תכולה |
|-------|--------|

| `/admin/bot/leads` | טבלת לידים — מסנן לפי סטטוס, תעשייה, חבילה |
| `/admin/bot/conversations` | היסטוריית שיחות — לפי ליד |
| `/admin/bot/rules` | עורך כללים (Phase 2) — ניהול ויזואלי |
| `/admin/bot/analytics` | אנליטיקס — funnel, drop-off, conversion rates |
| `/admin/bot/settings` | הגדרות — Webhooks, שעות פעילות, opt-out |

### 8.4 מה **לא** אפשרי (מגבלות Blaster)

| פעולה | אפשרי? | הסבר |
|--------|---------|-------|
| קבלת webhook כשליד כותב | ✅ כן | Blaster `SendWebhook` |
| שליחת הודעה מהבקנד שלנו | ⚠️ מוגבל | רק דרך API של Blaster (לא מתועד רשמית) |
| עדכון כללים מרחוק | ❌ לא | צריך לעדכן את קובץ .rules ישירות |
| קריאת שיחות | ⚠️ מוגבל | רק דרך ייצוא Excel |
| שליטה במשתנים | ✅ חלקית | Webhook יכול לקבל משתנים, אבל לא לשנות אותם בזמן אמת |

### 8.5 Phase 2: עורך כללים בתוך MISRAD AI

בשלב שני, אפשר לבנות:
1. **עורך ויזואלי** — drag & drop של כללים, כמו Flowchart
2. **ייצוא ל-.rules** — לחיצה על כפתור → מייצר קובץ .rules
3. **סנכרון** — כל שינוי → ייצוא אוטומטי → Blaster

זה מבטל את הצורך לערוך את הסקריפטים ידנית.
